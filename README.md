# HugoDesk

把 `Hugo + GitHub Pages` 这套原本偏终端、偏配置文件、偏分散工具协作的工作链路，收进一个可以长期日用的 macOS 桌面工作台。

它不是一个只负责写 Markdown 的编辑器，也不是一个给 Git 推送包一层 UI 的壳。HugoDesk 更关注的是整条链路是否顺手：写作时结构清不清楚，内容资源有没有放对位置，Front Matter 能不能稳定维护，多语言目录和页面引用会不会埋雷，发布时到底卡在本地、GitHub Actions，还是 Pages 配置。

如果你还想继续用 Hugo，但不想每天在终端、配置文件、仓库页面、日志页面和编辑器之间来回切，HugoDesk 的目标就是把这些上下文合成一个完整工作面。

---

## 产品定位

HugoDesk 是一个面向 Hugo 项目的桌面软件，核心目标有三件事：

- 让内容创作和 Hugo 结构保持一致，而不是写完再回头补结构。
- 让项目配置、主题参数、页面资源、多语言内容和发布状态放进同一个工作上下文。
- 让 GitHub Pages 的发布链路变成“可检查、可定位、可修复”，而不是只会失败提示。

它适合的是对 Hugo 有真实日常使用需求的人，而不是只想偶尔改一篇 Markdown 的场景。

---

## 适合谁

- 想继续用 Hugo，但不想把日常流程全部压在终端上的创作者。
- 使用 GitHub Pages 部署，希望把发布检查和排障做稳的人。
- 维护文档站、多语言站、内容结构比较复杂的博客或知识站的人。
- 想把写作、内容管理、配置编辑和发布操作收进一个桌面软件的人。

## 核心能力

### 1. 写作不是单文件编辑，而是内容结构管理

HugoDesk 的写作页不是只给你一块正文文本框。

它会直接围绕 Hugo 的内容模型展开：

- 左侧树形内容浏览器，识别单文件、页面包（Leaf Bundle）与栏目包（Branch Bundle）。
- 新建内容时可选择内容形态、Front Matter 格式与 archetype。
- 在窄窗口和常规窗口下都保持完整写作流程，不把核心信息挤成碎片。
- 正文区默认使用 `Vditor`，同时保留兼容模式，兼顾稳定性与输入效率。

这意味着软件理解的是“这是一篇怎样的 Hugo 内容”，而不只是“这是一份 Markdown 文本”。

### 2. Front Matter 真正进入可维护状态

HugoDesk 把一部分本来容易在纯文本里写乱的内容，转成可以长期维护的图形化界面：

- 支持 `TOML / YAML / JSON` 三种 Front Matter 格式。
- 支持结构化模式和原始模式切换，适合日常编辑和特殊字段微调两种场景。
- 支持标签、分类、关键词、别名、slug、URL、摘要、菜单项等常见内容字段。
- 支持动态 taxonomy、taxonomy 权重、`build`、`cascade.build` 等更偏 Hugo 原生的高级字段。

目标不是替代文本本身，而是让结构字段更稳定、更清楚、更不容易出错。

### 3. 页面资源工作流进入主线

很多 Hugo 项目真正难维护的，不是正文，而是资源。

HugoDesk 对这块的处理是直接按 Hugo 方式来：

- 支持页面资源导入到当前页面包。
- 支持资源列表浏览、插入正文、改名、移动、删除。
- 同时保留 `static` 目录图片工作流，兼容旧项目和简单站点。
- 在页面包和普通单文件之间做区分，避免资源路径混乱。

这会让“文章正文”和“页面资源目录”真正形成一个整体，而不是靠人工记忆维护。

### 4. 多语言内容与站内引用不再靠肉眼排查

如果站点不是单语单目录，很多问题不会出在编辑器，而会出在内容组织层。

HugoDesk 把这些能力收进写作页：

- 基于 `contentDir` 的内容工作区切换。
- 翻译副本创建。
- 缺失翻译诊断。
- `ref / relref` 插入。
- 标题锚点选择。
- 全工作区的失效引用检查。

这部分的价值很直接：不是把功能堆进界面，而是把原本最容易漏掉的结构性错误尽早暴露出来。

### 5. shortcode 和高级 Hugo 内容能力可视化

HugoDesk 不回避 Hugo 的高级能力，而是尽量把这些能力做成可发现、可选择、可插入：

- shortcode 扫描。
- 参数提示与原始参数补充。
- 菜单 Front Matter 和菜单树预览。
- `sectionPagesMenu`。
- `build` 与 `cascade.build`。

对于中后期的 Hugo 项目来说，这些能力不是锦上添花，而是内容组织效率的核心部分。

### 6. 主题设置和项目配置不再散落在文件里

HugoDesk 会把主题和项目设置纳入同一个桌面上下文：

- 自动检测本地主题。
- 切换当前主题。
- 维护常见主题参数、搜索、评论、数学公式、社交链接等配置项。
- 配置 GitHub 远端、发布分支、令牌、Hugo 可执行文件。
- 支持本地配置包导入导出，方便迁移和备份。

这类能力的重点是降低“知道该改哪里，但每次都得重新找”的认知成本。

### 7. 发布不是一个按钮，而是一套可验证的链路

HugoDesk 的发布页不是把 `git push` 换成图形按钮这么简单。

它更重视发布前后的确定性：

- 发布前检查。
- GitHub 连通性检测。
- GitHub Pages 来源检查。
- Workflow 状态同步。
- 本地时间显示运行时间。
- 原始日志与结构化日志查看。
- 一键发布、提交并推送。

遇到问题时，重点不是“再试一次”，而是尽量把失败点显示出来，让定位和修复更快。

## 页面结构

- `项目`：项目根目录、Git 远程、令牌、Hugo 工具、配置包。
- `写作`：内容树、写作工作台、AI 写作、正文编辑、Front Matter、页面资源、引用、短代码。
- `主题设置`：主题检测、切换与主题参数编辑。
- `AI 设置`：模型地址、模型名、API Key。
- `发布`：预检、Workflow、Pages、日志与修复操作。

## 设计思路

### 1. 尊重 Hugo 原生结构

HugoDesk 的前提不是“把 Hugo 藏起来”，而是“把 Hugo 原本需要理解的结构变得更容易操作”。

所以它会优先做这些事情：

- 页面包和栏目包的区分。
- 页面资源的真实落盘。
- 多语言内容目录的切换和诊断。
- Hugo 原生 Front Matter 能力的图形化。
- `ref / relref`、shortcode、menu、build 等内容能力的直接支持。

也就是说，它是更可用的 Hugo 工具，而不是一个脱离 Hugo 现实模型的内容平台。

### 2. 把问题暴露在前面，而不是埋到发布后

真正影响稳定性的，通常不是编辑器少一个按钮，而是这些问题：

- 结构不一致。
- 资源路径放错。
- 翻译副本缺失。
- 页面引用失效。
- Pages 来源不对。
- Workflow 状态滞后。
- 远端配置和本地理解不一致。

所以 HugoDesk 会把很多精力放在诊断、提示、修复入口和状态同步上，而不是只追求表面上的“轻量”。

### 3. 让日常使用成立，而不是只在演示里成立

很多工具在功能表上看起来很完整，但一旦进入持续写作场景，就会暴露出上下文来回切换的问题。

HugoDesk 的设计方向更偏实际生产流程：

- 新建内容后立即进入可继续编辑状态。
- 内容树、结构信息、资源和正文在一个上下文里连续工作。
- 窄窗口场景下也尽量保持操作顺序合理。
- 文档、说明和产物尽量跟随版本一起更新。

核心不是功能多，而是链路闭合。

---

## AI 在这里扮演什么角色

HugoDesk 里的 AI 能力不是替代写作，而是作为素材生成和初稿辅助：

- 接收文字素材。
- 接收链接素材。
- 混合输入后生成可直接追加到正文的 Markdown。
- 与正文编辑器分区，避免把 AI 区变成主界面噪音。

它更像是写作工作流中的一个辅助台，而不是整个产品的中心。

---

## 为什么不是“通用博客编辑器”

因为这个产品一开始就不是面向所有内容平台，而是面向 Hugo 项目本身。

它会优先考虑：

- 文件结构是否真实。
- 仓库是否可维护。
- 生成逻辑是否和 Hugo 一致。
- 发布链路是否能对上 GitHub Pages。

这也是它和很多“好写但不落地”的编辑器最大的不一样。
