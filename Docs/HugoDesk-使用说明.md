# HugoDesk 使用说明

版本：`0.4.0`

适用范围：
- Hugo 博客
- Hugo 文档站
- 多语言 Hugo 站点
- 需要在桌面软件里完成写作、资源管理、配置编辑与 GitHub Pages 发布的项目

---

## 1. 软件定位

HugoDesk 是一个面向 Hugo 项目的 macOS 桌面工具。

它的目标不是取代 Hugo 命令行，而是把下面这些高频操作集中到一个图形界面里：
- 编辑文章
- 管理 Front Matter
- 管理主题设置
- 导入图片和页面资源
- 检查 Hugo 项目结构
- 提交并推送到 GitHub
- 触发 GitHub Pages 工作流并查看状态

HugoDesk `0.4.0` 是当前正式版本。

这一版正式把 Hugo 的内容模型能力纳入主线，重点包括：
- 页面包（Page Bundle）
- 页面资源（Page Resources）
- archetype 创建内容
- 多语言内容目录
- 树形内容浏览器
- 翻译缺口诊断
- 菜单 Front Matter
- 菜单树预览与 `sectionPagesMenu`
- build 选项
- `cascade.build`
- `ref / relref`
- 标题锚点
- shortcode 参数提示

---

## 2. 进入软件前的准备

建议先准备好以下内容：

1. 一个已经初始化的 Hugo 项目根目录
2. 本地可用的 Hugo 可执行文件
3. 本地 Git 仓库
4. 对应的 GitHub 仓库
5. 用于发布的 GitHub Token
6. 如果要用 AI 写作，准备好 AI 接口地址、模型名和 API Key

如果你的项目已经能在终端里运行下面的命令，通常就适合交给 HugoDesk 管理：

```bash
hugo server
hugo build
git status
git push
```

---

## 3. 软件整体结构

软件主要分为 5 个区域：

1. 项目设置
- 配置 Hugo 路径、内容目录、远程仓库地址、Token 等

2. 写作
- 新建内容、编辑正文、管理 Front Matter、页面资源、短代码、站内引用

3. 主题设置
- 读取和编辑 `hugo.toml`
- 维护主题参数、社交链接、搜索、评论、数学公式等

4. AI 设置
- 配置 AI 接口地址、模型名与 API Key
- 为 AI 写作和后续 AI 能力提供调用参数

5. 发布
- 做发布前检查
- 提交并推送
- 检查 GitHub Pages 和 GitHub Actions 工作流状态

---

## 4. 初次使用建议流程

推荐按这个顺序完成首次配置：

1. 在“项目设置”中选择你的 Hugo 项目根目录
2. 确认 `content` 目录、Git 远程地址、发布分支设置正确
3. 在“远程地址与凭据”里保存 GitHub Token
4. 在“主题设置”中检查当前主题是否识别正确
5. 进入“写作”页面，新建一篇测试内容并保存
6. 去“发布”页面做一次完整预检
7. 再执行“一键发布”或“提交并推送”验证全流程

## 5. 写作页总览

当前版本的写作页已经改成了“左写右设”的结构：

### 左侧
- 内容列表
- 按目录分组显示文章或内容包

### 中间主区
- AI 写作
- 正文编辑器
- 保存与构建操作

### 右侧设置面板
分成 3 组：
- 基础信息
- Hugo 高级
- 资源与诊断

这样做的目的，是把真正高频的写作行为留在主区，把复杂的 Hugo 结构能力塞进右侧检查器，减少打断。

## 6. 写作工作台

写作页顶部的“写作工作台”包含两个部分。

### 6.1 内容工作区

这里对应 Hugo 的 `contentDir` 概念。

如果你的 `hugo.toml` 中配置了多语言或多内容目录，例如：

```toml
[languages.zh-cn]
contentDir = "content"

[languages.en]
contentDir = "content.en"
```

那么你可以在这里直接切换当前写作工作区。

切换后：
- 左侧内容列表会重载为对应目录下的内容
- 新建内容会写入当前工作区
- 站内引用候选页面也会基于当前工作区刷新

### 6.2 创建翻译副本

如果当前项目存在其他语言工作区，可以直接创建翻译副本。

它会：
- 复制当前文章的内容结构
- 保持相对路径结构一致
- 自动补上 `translationKey`（若原文为空则生成）

适合：
- 中文站翻译成英文站
- 主站与国际版同步维护

### 6.3 新建内容

支持三种内容形态：

1. 单文件
- 生成普通 Markdown 文件
- 适合普通博客文章

2. 页面包（Leaf Bundle）
- 生成 `index.md`
- 适合文章与资源文件一起维护
- 例如一篇文章带多张图、PDF、附件

3. 栏目包（Section Bundle）
- 生成 `_index.md`
- 适合栏目首页、文档目录页

还支持：
- 选择 Front Matter 格式：TOML / YAML / JSON
- 选择 archetype 模板类型

如果项目里存在可用的 archetype，软件会优先尝试用 Hugo 的模板创建内容；失败时会自动退回 HugoDesk 内部模板。

## 7. 正文编辑

### 7.1 默认编辑器

当前版本默认使用 `Vditor` 作为正文编辑器。

优点：
- 编辑体验更接近现代 Markdown 编辑器
- 内置预览能力
- 支持图片、脚注、表格、代码块等常见语法操作

如果你想对照稳定性，也可以切回“兼容模式”。

### 7.2 图片插入

正文区上方可以控制图片插入方式。

可选图片去向：

1. 自动
- 如果当前文章是页面包，则写入页面包
- 如果当前文章是普通单文件，则写入 `static/images/uploads`

2. 页面资源
- 强制导入到当前页面包
- 适合 Hugo 页面资源工作流

3. 静态目录
- 强制导入到 `static/images/uploads`
- 适合老项目或统一图片目录习惯

### 7.3 图片说明文字

`alt` 文本会写入 Markdown 图片语法中。

如果你不填，软件会尽量用文件名自动生成一个默认说明。

### 7.4 正文保存

正文区底部提供 3 个常用操作：
- 保存当前内容
- 保存并构建站点
- 删除当前内容

建议习惯：
- 普通写作时：只用“保存当前内容”
- 修改模板或资源链路时：用“保存并构建站点”快速验证

---

## 8. AI 写作

AI 写作模块位于正文编辑上方。

它不是普通润色按钮，而是一个“素材输入 -> 模型生成 -> 回写正文”的独立流程。

### 8.1 支持输入的内容
- 一段纯文字
- 一个或多个链接
- 文字 + 链接混合输入

### 8.2 工作方式
软件会：
1. 读取输入内容
2. 识别其中的可访问链接
3. 尝试抓取链接正文摘要
4. 把素材交给预设模型
5. 将返回的 Markdown 直接追加到正文编辑器

### 8.3 适用场景
- 把资料链接整理成文章初稿
- 对原文做二次改写
- 生成博客摘要、小结、引言
- 结合参考资料补全某个章节

## 9. 基础信息面板

右侧“基础信息”面板包含三块内容：
- 内容结构
- 页面信息
- 分类与摘要

### 9.1 内容结构

这里会显示：
- 当前页面类型
- Hugo 相对路径
- 实际内容文件路径
- 页面包根目录路径

你可以用它快速确认：
- 当前是不是页面包
- 资源应该放到哪里
- 当前页面在项目中的真实位置

### 9.2 页面信息

这个区域对应页面的 Front Matter 基础字段。

支持两种模式：

1. 结构化模式
- 适合日常编辑
- 以表单方式填写标题、日期、摘要、slug、url 等字段

2. 原始模式
- 适合高级用户
- 直接编辑原始 Front Matter 文本

支持的 Front Matter 格式：
- TOML
- YAML
- JSON

常用字段包括：
- 标题
- 日期
- 草稿
- 摘要
- slug
- url
- aliases
- translationKey
- 作者
- 封面路径
- 关键词
- 置顶
- 数学公式开关
- 搜索可见性
- 私有标记

### 9.3 分类与摘要

这里用于管理：
- `tags`
- `categories`
- 项目自定义 taxonomy

同时也能控制 Hugo 摘要行为：
- 自动摘要
- 使用 Front Matter 摘要
- 插入 `<!--more-->` 做手动分隔

如果你在写博客列表摘要，建议：
- 重要文章：手动写摘要
- 文档页：插入 `<!--more-->`
- 普通短文：使用自动摘要也可以

## 10. Hugo 高级面板

右侧“Hugo 高级”面板包含：
- 菜单与构建策略
- 站内引用与短代码

### 10.1 菜单入口

这里对应 `menus` Front Matter。

适合场景：
- 页面进入顶部导航
- 页面进入文档目录
- 自定义菜单排序

常见字段：
- 菜单名 `menuName`
- 显示名 `name`
- 标识 `identifier`
- 父级 `parent`
- 前后缀 `pre` / `post`
- 权重 `weight`

### 10.2 构建策略

这里对应 `build` Front Matter。

用于控制：
- 是否参与列表
- 是否单独渲染
- 页面资源是否发布

适合场景：
- 目录页
- 资料页
- 仅供 shortcode 调用的内容块
- 接近 headless 的内容结构

如果你不确定是否需要改这里，建议保持默认。

### 10.3 页面引用

这里可以可视化插入：
- `relref`
- `ref`

推荐优先使用：
- `relref`

因为它对 Hugo 内部内容结构变动更稳，尤其适合文档站。

### 10.4 短代码

软件会扫描：
- 项目本地 `layouts/shortcodes`
- 当前主题中的 `layouts/shortcodes`

你可以：
- 选择某个 shortcode
- 填写参数
- 决定插入为行内还是块级

这比手写语法更稳，尤其适合复杂参数场景。

---

## 11. 资源与诊断面板

右侧“资源与诊断”面板包含：
- 页面资源
- 引用诊断

### 11.1 页面资源

只有当当前内容是页面包时，这里才真正发挥作用。

可做的事：
- 导入资源到当前页面
- 查看当前页面包中的资源列表

适合管理：
- 图片
- PDF
- 音频
- 下载附件

### 11.2 引用诊断

这个区域会扫描当前工作区中的：
- `ref`
- `relref`

如果引用目标不存在，会在这里提示。

适合用于：
- 文档站内链检查
- 多目录迁移后的引用复查
- 发布前快速排错

---

## 12. 项目设置页

项目设置页主要负责：
- Hugo 可执行文件路径
- 内容目录路径
- 本地配置包
- 远程仓库地址
- GitHub Token
- AI 设置

### 12.1 Token 建议

软件目前支持两种 Token：
- Fine-grained Token
- Classic Token

建议：
- Git 推送可以先用 Fine-grained
- GitHub Pages / 部分 API 检测如果不稳定，补一个 Classic Token 更稳

### 12.2 本地配置包

软件会把一部分项目相关设置同步保存到本地配置包中，便于：
- 迁移项目
- 恢复设置
- 做独立备份

---

## 13. 主题设置页

主题设置页会读取当前主题与 `hugo.toml`，提供图形化配置。

已支持的方向包括：
- 基础站点信息
- 社交链接
- 搜索
- 评论
- 数学公式
- 图片资源
- 主题能力检测

如果当前主题能被检测到，软件会尽量只显示该主题需要的设置项。

---

## 14. 发布页

发布页主要负责：
- 发布前检查
- GitHub 连通性
- Hugo 构建
- 与远端同步
- 提交并推送
- GitHub Actions / Pages 状态查看

### 14.1 发布前检查建议

每次发布前，至少确认：
- 项目配置文件正常
- 本地配置包正常
- 推送地址正常
- 部署策略正常
- GitHub Token 正常
- Pages 来源正常
- Pages Workflow 正常
- 当前主题识别正常
- Git 冲突为零

### 14.2 推荐发布习惯

建议使用这个顺序：

1. 在写作页保存内容
2. 到发布页做完整预检
3. 先看是否有结构或权限错误
4. 再执行发布流程
5. 最后查看最新 workflow 状态是否同步成功

## 15. 推荐工作流

### 15.1 普通博客

推荐：
- 内容形态：单文件
- 图片去向：静态目录
- 摘要：Front Matter 或自动摘要
- 发布：保存 -> 发布页预检 -> 一键发布

### 15.2 图片较多的专栏文章

推荐：
- 内容形态：页面包（Leaf Bundle）
- 图片去向：页面资源
- 摘要：手动写摘要或 `<!--more-->`
- 正文中配合 shortcode 或页面资源引用

### 15.3 文档站

推荐：
- 栏目页使用栏目包（Section Bundle）
- 内容页使用页面包（Leaf Bundle）
- 菜单通过 `menus` 管理
- 站内链接优先使用 `relref`
- 经常查看“引用诊断”

### 15.4 多语言站点

推荐：
- 在 `hugo.toml` 中明确配置多个 `contentDir`
- 在写作页切换内容工作区
- 用“创建翻译副本”复制结构
- 用 `translationKey` 维护翻译组关系

## 16. 常见问题

### 16.1 为什么页面资源按钮不可用？
因为当前内容不是页面包。

解决方式：
- 新建“页面包（Leaf Bundle）”
- 或“栏目包（Section Bundle）”

### 16.2 为什么站内引用列表里没有页面？
通常是因为：
- 当前工作区为空
- 还没有保存内容
- 正在错误的 `contentDir` 中查看

### 16.3 为什么构建策略看起来很复杂？
因为它对应 Hugo 原生的 `build` 行为，本来就属于高级功能。

如果你只是写普通博客文章，可以保持默认，不需要改。

### 16.4 为什么短代码没有扫描到？
请检查：
- 项目是否存在 `layouts/shortcodes`
- 当前主题目录是否正确
- 主题名是否能被软件识别

### 16.5 为什么 GitHub Pages 检查失败，但 Git 推送还能成功？
因为 Pages API 权限和 Git 推送权限不是一回事。

常见原因：
- Token 权限不足
- 仓库 Pages 尚未启用
- Pages 来源还不是 GitHub Actions

## 17. 这版 0.4.0 的重点

`0.4.0` 最重要的不是多几个按钮，而是软件已经开始真正理解 Hugo 的内容结构。

这意味着：
- 它不再只是 Markdown 编辑器
- 它开始接近一个 Hugo 内容工作台

当前版本最值得你重点体验的部分：
- 页面包与页面资源
- 多语言工作区切换
- 翻译副本创建
- 菜单与 build 策略
- `ref / relref` 与 shortcode 插入
- 失效引用诊断

---

## 18. 建议你的测试顺序

建议你按下面顺序测试：

1. 新建一篇普通单文件文章
2. 再新建一篇页面包文章并导入图片
3. 插入一个 `relref` 引用
4. 插入一个 shortcode
5. 设置一个菜单入口和 build 策略
6. 如果项目有多语言目录，再创建一个翻译副本
7. 最后去发布页做一次完整预检和构建

这样能最快看出这版功能是否符合你的真实 Hugo 工作流。
